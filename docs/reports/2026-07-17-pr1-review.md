# PR 1 Review — Report Data Pipeline + React SPA (Initial Commit)

**Date:** 2026-07-17
**PR:** [#1 — feat: report data pipeline (uv generator) + React SPA](https://github.com/kerryhatcher/naacp-report/pull/1)
**Method:** Four-agent tiered review — Haiku (config/CI hygiene), Sonnet ×2 (generator code, SPA code), Opus (architecture) — with lead verification of all load-bearing claims. One Haiku finding was discarded as a false positive after verification (see Appendix).

---

## Overall verdict

**The bones are good.** The seams are in the right places: clean file-based handoff between generator and web, generated output correctly gitignored (committed-vs-generated drift is structurally impossible), sanitization done properly on both sides, least-privilege CI. Above-average discipline for an initial commit. One architectural decision is worth revisiting now while it's cheap, plus a handful of real bugs in the generator.

---

## 1. The big one: client-rendered SPA is misaligned with the mission

> **Resolved 2026-07-17:** Decision made to pivot to Astro SSG with React islands, as a follow-up PR after PR 1 merges. See `docs/decisions/0001-astro-ssg-with-react-islands.md`.

A public civic report wants search discoverability, reliable direct links, fast first paint on cheap phones, and longevity. A client-rendered React SPA on GitHub Pages gives none of those by default — it ships an empty HTML shell, and the pending `404.html` deep-link shim and base-path juggling in the pre-deploy checklist are symptoms of fighting the platform.

All routes are known at build time (159 county slugs + 3 section pages) and the data is already static: this is a textbook static-site-generation case. **Today, with only 2 real routes wired, is the cheapest this pivot will ever be** — the cost climbs with every county page and inbound link.

Options in ascending effort:
1. Prerender known routes post-build (puppeteer crawl / react-snap style — react-snap itself is flaky on React 19).
2. Adopt Vike (`vite-plugin-ssr`) for prerendering within the existing React app.
3. Adopt Astro with islands, still fed by the same Python JSON.

The generator, the JSON contract, and the deploy workflow all survive any of these untouched — a credit to the boundary design.

### Architecture decisions to KEEP
- **Static list+detail JSON handoff** — scales fine to 159 counties: one ~20KB list file, details fetched on navigation. Rejected alternatives correctly rejected (merged JSON forces all `body_html` down the wire; bundle-time import couples data to code).
- **Two toolchains (Python + Node)** — Python earns its place once turnout/demographics ETL arrives; that's pandas/geopandas work, not Node work. The two-runtime tax is front-loaded and cheap.
- **`build.py` orchestration seam** — each builder owns inputs → writes JSON; genuinely extensible.
- **Gitignored `web/public/data/`** — regenerate in CI; drift impossible.

### WATCH items from the architecture review
- `public/` files are copied verbatim, not content-hashed; Pages serves `Cache-Control: max-age=600`, so a data-only redeploy can serve ~10-min-stale JSON. Wants a cache-bust (build-hash query param or manifest) once content updates matter.
- No preview environments — once stakeholders review draft content, per-PR preview deploys will be wanted.
- A CSV→md bootstrap script will be needed so nobody hand-types 159 county files.

## 2. Real bugs in the generator (high severity — all verified by reproduction)

1. **Slugs are raw filename stems, and they're load-bearing for URLs** (`generator/builders/county_boards.py:15`). Georgia has counties named "Ben Hill" and "Jeff Davis" — a file named `Ben Hill.md` produces a URL-unsafe slug and a broken route. Fix: real slugify + `^[a-z0-9-]+$` build-time assertion + slug-collision detection (duplicate slugs currently silently overwrite each other).
2. **Stale output is never cleaned up** (`county_boards.py:24-45`). Delete a county's `.md` and rebuild — its detail JSON stays on disk and gets deployed forever. Fix: wipe the output dir before writing.
3. **No validation, no atomicity** (`county_boards.py:12-21`, `build.py`). `members: five` (a string) flows straight into JSON despite the TS type saying `number | null`; a missing/misspelled frontmatter key silently becomes `null` (renders as "—", nobody notices the data is wrong); malformed YAML crashes mid-loop leaving mixed old/new output with no file named in the error. **One fix covers all three: a pydantic model as the output contract** — validates frontmatter at build time, fails loudly naming the offending file, and later doubles as the JSON-Schema source for TS codegen. Retrofitting after 159 hand-authored files means auditing all 159.

### Generator — medium/low
- `markdown.markdown()` called with no extensions — tables and fenced code blocks render as literal text (`county_boards.py:20`). Enable `extensions=["tables", "fenced_code"]` or document the restriction.
- `OUTPUT_DIR` computed via triple `.parent`, duplicated in three builder files — moving a file breaks the path silently. Centralize.
- `test_smoke.py` asserts only that the package imports — near-zero value.
- Sanitization verified **solid**: nh3 strips `<script>`/`onerror`/`javascript:`; non-HTML fields render as auto-escaped JSX text; only `body_html` hits `dangerouslySetInnerHTML`, double-sanitized.

### Three most valuable missing generator tests
1. Rebuild-after-removal leaves no stale output (currently fails — the bug above).
2. Malformed/missing-field frontmatter fails loudly, naming the offending file.
3. Multi-county build with a spaced name ("Ben Hill") produces URL-safe, non-colliding slugs.

## 3. CI gap (high leverage, trivial fix)

`deploy.yml` runs **no tests, no lint, no typecheck** before building and shipping — it can publish broken data or a white screen unnoticed. Add `uv run pytest` + `ruff check` + `tsc --noEmit` as a gate before the build steps. Related: the web app has no test framework at all; a single smoke test asserting `/counties` renders a row would prevent shipping a blank public report. The deferred cost of skipping this — a publicly broken report — is reputational.

## 4. SPA findings (medium — fast-follows, not blockers)

- **404s look like hangs**: TanStack Query default retries (3× with backoff) apply to 404s on static JSON that will never appear; an unknown slug shows "Loading…" for seconds before erroring (`web/src/main.tsx:8`). Set `retry: false` and `staleTime: Infinity` — the data is immutable build output. Default `refetchOnWindowFocus` also refetches static data pointlessly.
- **No catch-all route** (`web/src/App.tsx:23-29`): unmatched URLs render just the nav over a blank body. Add `<Route path="*" element={<NotFound />} />`.
- **Accessibility**: no `<main>` landmark; no focus management or `document.title` updates on route change (declarative `BrowserRouter` — `<ScrollRestoration>` needs a data router). For a public report this matters.
- **Lint blind spot**: `.oxlintrc.json`'s explicit `plugins` array *replaces* oxlint's defaults (drops `unicorn`) and never enables `jsx-a11y` — so the a11y gaps above get no lint signal. Turn `jsx-a11y` on.
- `response.json() as Promise<T>` is an unvalidated cast — malformed JSON renders a subtly blank page, not a useful error (`web/src/lib/api.ts:12`). Acceptable for now; noted.
- Redundant `data!` non-null assertions after narrowing in `CountyDetail.tsx`; `CountiesList.tsx:26` uses `data!.map` with no guard at all.
- Base-URL checklist status verified: `api.ts` BASE_URL-relative fetches **done**; Vite `base` and `404.html` fallback **still outstanding** (deploy correctly gated to `workflow_dispatch` meanwhile).
- Types verified: `web/src/types/county.ts` matches generator output field-for-field today.

## 5. Judgment calls to make deliberately (not bugs)

- **Double sanitization** (nh3 at build + DOMPurify at render): defensible defense-in-depth per the PR's explicit decision, but two sanitizer configs can silently disagree, and DOMPurify costs ~20KB client-side. Either document and keep configs aligned, or declare nh3 the single authoritative boundary.
- **The list+detail pattern won't fit turnout/demographics.** Those are tabular/cross-tab data with no `body_html` — the orchestration seam holds, but don't let docs promise all collections share the county-boards JSON shape. The `[]` stubs commit to nothing, so this costs only a doc correction now.

---

## Ranked recommendations (by cost-to-fix-later)

1. **Decide the rendering strategy now** — prerender known routes or pivot to SSG (Vike/Astro). The only decision whose reversal cost compounds with every county page and inbound link.
2. **Add a pydantic contract model + slugify/collision/cleanup fixes in the generator.** Every county authored without a schema compounds silent-bad-data risk.
3. **Add a CI test/lint/typecheck gate + one web smoke test.** Trivial code; reputational insurance.

These interact: if you pivot to SSG (1), the JSON contract and frontmatter model (2) become more load-bearing because the build consumes them at render time — one more reason to settle both before real turnout/demographics data forces the shapes.
